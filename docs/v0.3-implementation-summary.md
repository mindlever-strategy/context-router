# Context Router v0.3 Implementation Summary

**Implementation date:** 2026-07-21  
**Status:** Implemented and locally verified

## Overview

Context Router v0.3 introduces an approachable local development path that no
longer requires Docker, PostgreSQL, environment configuration, or manual
database migrations. SQLite is now the default local store, while PostgreSQL
remains available as an explicit production-oriented option.

The release also adds a workflow-scoped TypeScript API, packaged local server
startup, CLI diagnostics, two new MCP tools, and a runnable quickstart.

## Storage and startup

- Added separate Prisma schemas and generated clients for SQLite and
  PostgreSQL.
- Added automatic storage detection:
  - `file:` and `sqlite:` URLs select SQLite.
  - `postgres:` and `postgresql:` URLs select PostgreSQL.
  - No `DATABASE_URL` selects SQLite automatically.
- Added optional `STORAGE_ENGINE` validation. A mismatch between the explicit
  engine and `DATABASE_URL` produces a startup error.
- Added default per-user SQLite locations:
  - Windows: `%LOCALAPPDATA%\context-router\context-router.db`
  - macOS: `~/Library/Application Support/context-router/context-router.db`
  - Linux: `$XDG_DATA_HOME/context-router/context-router.db`, falling back to
    `~/.local/share/context-router/context-router.db`
- Added `CONTEXT_ROUTER_DATA_DIR` as an optional SQLite location override.
- Added packaged, transactional SQLite migrations with migration checksums.
- Enabled SQLite foreign keys, WAL journal mode, and a five-second busy timeout.
- Retained explicit Prisma migrations for PostgreSQL. Context Router does not
  automatically migrate data between SQLite and PostgreSQL.

## MCP server changes

The public MCP surface increased from 27 to 29 tools.

### `workspace_ensure`

Gets or creates one stable workspace using a normalized name. Names are
trimmed, Unicode-normalized with NFKC, and compared in lowercase.

Existing legacy workspaces are adopted when exactly one name matches. Multiple
legacy matches return `WORKSPACE_NAME_AMBIGUOUS` instead of selecting one
silently.

### `router_status`

Returns:

- Context Router version;
- active storage engine;
- the SQLite file location, when applicable;
- owner-scoped workspace, workflow, running-workflow, and checkpoint totals;
- the ten most recent workflows.

PostgreSQL connection strings and credentials are never included in status
responses.

## TypeScript SDK changes

All existing explicit workspace and workflow-ID APIs remain available.

The following additive APIs were introduced:

```typescript
const router = await ContextRouter.local();
const flow = await router.start('Research');

await flow.set('findings', { answer: 42 });
const handoff = await flow.handoff({ keys: ['findings'] });

console.log(handoff.summary);

await flow.complete();
await router.close();
```

### `ContextRouter.local()`

- Resolves the installed `@context-router/mcp-server` package directly.
- Starts it with the current Node executable.
- Does not invoke a shell, `npx`, or the network.
- Supports optional `dataDir`, `databaseUrl`, and `ownerId` settings.

### `WorkflowSession`

`router.start(workspaceName)` ensures the named workspace, creates a fresh
workflow, and returns a scoped session with:

- `set()`;
- `get()` and `getMany()`;
- `checkpoint()`;
- `handoff()`;
- `complete()`;
- `fail()`.

Sessions delegate to the existing explicit API and do not silently complete or
delete workflows.

### Other additions

- `router.status()` calls the new `router_status` MCP tool.
- `router.close()` is a friendly alias for `disconnect()`.
- `router.discoverTools()` supports CLI health diagnostics.
- The SDK now installs a compatible MCP server as a runtime dependency.

## CLI commands

Installing `@context-router/sdk` now provides a `context-router` executable.

```bash
context-router doctor
context-router status
```

Both commands accept `--json` and `--data-dir <path>`.

`doctor` verifies:

- Node.js 20 or newer;
- packaged server resolution and startup;
- SQLite initialization or configured database access;
- storage-directory write access;
- MCP handshake and discovery of all 29 tools.

CLI exit codes are stable:

- `0`: successful or healthy;
- `1`: runtime or health failure;
- `2`: invalid CLI usage.

## Runtime and packaging

- Updated the root, server, and SDK Node.js requirement from 24+ to 20+.
- Updated the TypeScript compilation target to ES2022.
- Updated package versions to `0.3.0`.
- Added the SDK-provided `context-router` binary.
- Included both generated Prisma clients and SQLite migrations in the server
  package.
- Added a local package smoke test that verifies persistence across restarts.

## Documentation and examples

- Replaced the Docker-first README quickstart with a SQLite-first SDK example.
- Added `examples/simple-pipeline.ts`.
- Updated the MCP API documentation for 29 tools.
- Updated Cursor and contributor setup instructions for the zero-configuration
  path.
- Updated the adoptability strategy to distinguish completed work from future
  hypotheses.
- Updated the roadmap and changelog for v0.3.

## CI coverage

The CI definition now includes:

- validation on Node.js 20, 22, and 24;
- SQLite local-runtime checks on Windows, macOS, and Linux;
- PostgreSQL 16 migrations and integration tests;
- formatting, type checking, tests, builds, MCP discovery, and package dry-run
  checks.

## Verification performed

The implementation was verified with:

- 32 passing automated tests;
- successful TypeScript type checking;
- successful Prettier formatting validation;
- successful generation of both Prisma clients;
- successful server and SDK builds;
- successful discovery of all 29 MCP tools;
- successful SQLite persistence across process restarts;
- successful `doctor` and `status` CLI checks;
- successful npm dry-run packaging for the server and SDK;
- successful installation of both tarballs in an isolated consumer project;
- a successful quickstart executed from the installed SDK tarball;
- zero vulnerabilities reported by npm during dependency installation.

Docker was not available in the local implementation environment, so the
PostgreSQL integration suite could not be executed locally. PostgreSQL types,
schema generation, migrations, and the dedicated CI integration job remain in
place.

## Deliberately deferred

The following work remains outside v0.3:

- automatic SQLite-to-PostgreSQL data migration;
- first-class Python SDK;
- web playground or full workflow dashboard;
- LangGraph and CrewAI adapters;
- remote MCP transport;
- authentication and hosted multi-tenancy;
- managed deployment and billing.
